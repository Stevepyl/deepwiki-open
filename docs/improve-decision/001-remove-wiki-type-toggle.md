# ADR-001: Remove Wiki Generation Mode Toggle (Comprehensive vs Concise)

**Status**: Proposed
**Date**: 2026-04-19
**Author**: SteveLiu

---

## Context

DeepWiki-Open currently exposes a "wiki type" toggle in the UI (`isComprehensiveView`):

- **Comprehensive** (default): 8-12 pages with a two-level sections tree. Richer structure but more complex prompt engineering.
- **Concise**: 4-6 pages, flat list, no sections. Simpler but produces a lower-value output.

The toggle exists in three places:
- `src/components/WikiTypeSelector.tsx` — primary UI control
- `src/components/ConfigurationModal.tsx` — embedded in settings modal
- `src/app/page.tsx:131` — state management (`useState<boolean>(true)`)

The prompt branching exists in:
- `src/app/[owner]/[repo]/page.tsx:746-832` — two separate XML schema variants in the planner prompt
- Cache keys are segregated: `deepwiki_cache_..._comprehensive` vs `..._concise`

**With the Path B agent refactor (subtask 12)**, the planner is now a proper agent that uses `glob`/`ls`/`read` tools. The backend `WIKI_PLANNER_SYSTEM_PROMPT` already implements both modes via the `comprehensive_instruction` variable. However, maintaining two prompt branches long-term doubles the surface area for prompt engineering bugs and A/B testing complexity.

---

## Decision

**Remove the wiki type toggle entirely in a future release.** Default to comprehensive mode.

The rationale:
1. **Usage data**: The UI defaults to comprehensive (`useState(true)`), strongly suggesting that is the expected experience.
2. **Maintenance burden**: Two prompt branches (concise + comprehensive) in both the old frontend prompts and the new backend prompts doubles the testing surface.
3. **Agent quality**: The comprehensive output (8-12 pages, sections tree) is strictly more informative than concise (4-6 pages). The extra pages are generated by separate agent loop invocations, so there is no single-request cost penalty — only total session time increases.
4. **UX simplification**: Removing the toggle reduces cognitive overhead for new users.

---

## Current Scope (Subtask 12)

This ADR describes a **future** change. The current implementation (subtask 12 — Path B backend) retains the `comprehensive: bool` field in `AgentWikiRequest` and the backend `WIKI_PLANNER_SYSTEM_PROMPT` supports both modes. No frontend changes are made in this subtask.

| Step | Action | Timing |
|------|--------|--------|

---

## Consequences

**Positive**:
- Single prompt branch for backend wiki generation — easier to tune and test.
- Simplified frontend state management.
- Users always get the highest-quality output by default.

**Negative**:
- Users who preferred quick concise wikis lose that option. Mitigation: the comprehensive wiki always includes an "Overview" or "Getting Started" page that serves as a fast entry point.
- Cache invalidation: existing `..._concise` cache entries become orphaned. Mitigation: cache pruning on upgrade, or accept stale entries (they remain readable).

---

## Dependencies

- **Subtask 11** (Ask component agent mode integration): may reference `isComprehensiveView` — audit before removal.
- **Subtask 13** (Wiki frontend WebSocket adapter): the new frontend will connect to `/ws/agent-wiki` and may need to stop passing the `comprehensive` parameter or always pass `true`.
- **Wiki cache schema**: if we remove the mode from cache keys, existing caches must be migrated or invalidated.

---

## Related Decisions

- This ADR does **not** remove the backend `comprehensive` field immediately, preserving API backward compatibility during the transition period.
- The `concise` prompt variant in the old `/ws/chat` flow (`src/app/[owner]/[repo]/page.tsx:796-819`) is out of scope — it will be removed when the old flow is deprecated as part of the broader Path B migration.
